# mini-react PR 리뷰 가이드

## PR 체크리스트

- [ ] **성능**
  - 리렌더링/재귀 렌더 경로가 불필요한 O(n^2) 패턴을 만들지 않는가?
  - 문자열/배열 children 처리에서 반복 할당을 과도하게 늘리지 않았는가?
  - 렌더 경로에서 `container`를 반복적으로 완전 교체하는 방식인지 부분 업데이트 가능한지 정의했는가?
- [ ] **동시성**
  - 상태/렌더 동시 호출 시 큐 순서 보장(예: batched update 정책)을 고려했는가?
  - 여러 렌더 요청이 충돌할 때 마지막 요청이 최종 렌더에 반영되는지 정합성이 확보됐는가?
  - cleanup/dispose 경로가 렌더 취소/언마운트 상황에서 안전한가?
- [ ] **타입**
  - `VNode`, `Hook`, `Renderer` 타입이 사용 지점에서 좁혀지는지 확인.
  - `mini-react/src/index.ts` 공개 타입이 최소 1개라도 의도치 않게 확장/축소되었는가?
  - `tsconfig` 경로/빌드가 실제 타입 체크에서 깨지지 않는지 확인.
- [ ] **공개 API**
  - `packages/mini-react/src/index.ts`만 공개 export에 포함되는지 확인.
  - `src/**/internal.ts` 또는 `internal` 명명 모듈이 export되거나 문서화되지 않는지 확인.
- [ ] **테스트**
  - 이 커밋에 맞는 최소 `check:*` 경로(빌드/테스트/플레이그라운드) 실행 내역이 있는가?
  - 단위 테스트가 없을 경우, 왜 지금은 불필요한지 2줄로 설명되었는가?
  - 회귀 위험이 큰 구간(렌더링, hooks, createRoot, 타입 선언)은 적어도 하나 이상의 회귀 테스트 계획이 존재하는가?
- [ ] **호환성**
  - Node/브라우저/ESM 빌드 대상에서 import 호환성이 보장되는가?
  - 기존 사용 코드에 대한 breaking change 알림이 PR 설명에 명확한지?

## 리뷰 코멘트 예시

- `성능`: “렌더 함수가 현재 입력 전체를 순회해 DOM을 덮어쓰고 있습니다. children 길이가 크면 frame-drop이 발생할 수 있어 key 기반 병합 또는 최소 diff 전략이 필요합니다.”
- `동시성`: “`setState` 호출 직후 즉시 `render`를 실행하면 호출 순서 의존 버그가 생깁니다. 렌더 큐를 한 번에 압축하는 구조가 필요합니다.”
- `타입`: “공개 타입 변경이 번들 타입 선언에 반영되지 않습니다. `dist/index.d.ts`를 기준으로 consumer breakage 여부를 확인해 주세요.”
- `공개 API`: “`internal` 유틸이 재노출되면 번들 안정성보다 내부 리팩터링이 어렵습니다. Public Surface를 `src/index.ts`로 제한해 주세요.”
- `테스트`: “이 로직은 경계 조건(컨테이너 null/children null)에서 버그 가능성이 큽니다. 최소 1개 단위 테스트로 보강해 주세요.”
- `호환성`: “`render` 기본 동작이 변경되면 렌더링 타이밍 가정이 달라집니다. 기존 playground 동작을 기준으로 브라우저 빌드 확인이 선행되어야 합니다.”

## 리뷰어용 정렬 규칙

- 큰 API 변경은 먼저 문서(architecture/testing/pr-review) 갱신 여부를 확인한다.
- 리뷰 코멘트는 “원인 → 위험 → 제안” 순으로 작성한다.
- 구현이 완성되지 않은 TODO는 PR 본문에서 다음 커밋에서 해결할 항목으로 체크할 것.
